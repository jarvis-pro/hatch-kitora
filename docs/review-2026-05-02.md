# Kitora v0.9.0 工程复盘报告

> 评审日期：2026-05-02 ｜ 评审人：CTO 视角 ｜ 范围：代码质量、工程设计、文档同步

---

## 一、总体评估

Kitora 已经不是一个"模板"，而是一个**完成度极高、RFC 驱动**的 SaaS 基础平台：8 个 RFC 全部落地（多 region、多租户、SSO/SCIM、WebAuthn、GDPR、CN 区计费、后台 jobs），268 个 TS/TSX 文件 ≈ 28k 行，单测 + e2e + OpenAPI lint + egress audit 一应俱全。结构清晰、注释详尽、对边界（`server-only` / `'use server'` / `'use client'`）的把控明显在线。

但也已经走到了**第二曲线的拐点**——继续堆 RFC 不如花两周做一次"整理与收口"。本次评审发现 3 处 P0、若干 P1/P2 问题，集中在**重复表达、状态机膨胀、文档漂移**三个方向，下文按"代码质量 → 工程设计 → 文档同步"分章给出证据与改进建议。

---

## 二、代码质量与编写风格

### P0 — 必须立刻处理

1. **Stripe Webhook 幂等检查与处理顺序存在竞态**
   `src/app/api/stripe/webhook/route.ts` 先 `dispatch()` 再 `prisma.stripeEvent.create()` 写幂等行的顺序，理论上在两条同 event 的并发请求中可能双跑业务。建议改为"先尝试 `create`（依赖 PK 唯一）→ 命中 P2002 就 200 短路 → 否则进 dispatch"，把 DB 当成天然互斥锁。CN `BillingEvent` 路径同理需要复核。

2. **`api-auth.authenticateBearer` 的 `lastUsedAt` 副作用静默吞错**
   `src/lib/api-auth.ts` 在更新 `lastUsedAt` 时 `.catch(() => {})`，连接池耗尽 / DB hiccup 时不会冒泡也不写日志，影响审计与失活策略。建议至少 `logger.warn` 一次，或挪到 `BackgroundJob` 异步异步更新（这正好是 0.9 已经引入的能力）。

### P1 — 一般

3. **`src/lib/auth/index.ts` 的 jwt callback 已经太胖**
   一个回调里塞了：device session bump、`sessionVersion` 校验、`tfa_pending` 三段状态切换、`status / region` 同步。任何新增状态都会扎进这块。建议把"状态机"拆成 `evaluateAuthState({ token, freshUser })` 纯函数，回调只做 IO（DB 读、cookie 写）。

4. **`src/services/jobs/registry.ts` 的 `as unknown as AnyJobDefinition`**
   类型擦除靠注释保护，未来谁加一个 handler 但漏掉 `payloadSchema` 不会被编译期抓出。建议用 `satisfies JobDefinition<infer P, infer R>` 收窄；或在 `defineJob` 注册阶段对一个最小 dummy payload 跑一次 schema parse 抽烟雾测试。

5. **中英文注释混排 + 长篇大论**
   `src/lib/db.ts` 等核心文件大块中文段落注释对外协作不够友好，且同一文件里既有 `// 注释` 也有 `/** JSDoc */` 风格。建议统一英文 + JSDoc，业务背景留给 RFC。

6. **`src/app/api/auth/sso/callback/route.ts` 用手工 cookie 字符串解析**
   走 `cookies()` API 即可，免自己处理 RFC 6265 边角。

### P2 — 轻微

7. 多个 `Promise.all` 邮件发送（`src/services/webhooks/cron.ts` 等）建议改 `Promise.allSettled` 统一吞结果再聚合错误，避免一封失败影响整体收件。
8. `prisma.findUnique({ where: { email_region: ... } })` 跨多个 callsite 出现 5+ 次，可包装一个 `findUserInRegion(email)` 减小改动半径。
9. 单测覆盖**严重不均**：`src/services/jobs/*.test.ts`、`src/lib/auth/2fa-crypto*` 有；但 `src/services/billing/provider/{alipay,wechat}.ts`、`src/services/data-export/builder.ts`、`src/services/sso/*`、`src/services/webhooks/*` 完全无单测，全靠 e2e 兜——这三块是支付/SSO/外发关键面，建议补 happy-path + 错误码 2 个 case 即可拦住绝大多数回归。

## 三、工程设计合理性

### P0 — 必须立刻处理

1. **`vercel.json` 与 `runWorkerTick` 默认预算的隐式假设**
   `src/app/api/jobs/tick/route.ts` 用 `maxDuration = 60` 假定了 Vercel Pro，Hobby 计划的 10s 限制下只能跑 `{ budgetMs: 8000, batchSize: 1 }`。当前 route 自身**没有探测**计划，部署到 Hobby 即"静默部分失败 + lockedBy 卡死"。建议路由入口加一行 `process.env.VERCEL === '1' && !process.env.VERCEL_PRO` 检测短路，或直接读 `VERCEL_DEPLOYMENT_ID` + 判断超时回写。

2. **`vercel.json` 的 cron schedule 与设计意图不一致**
   `vercel.json:6` = `"0 0 * * *"`（每天 UTC 00:00），但 `docs/deploy/global.md:83` 与 `CHANGELOG.md` 都写 `* * * * *`（每分钟）。Schedule 本应是 jobs infra 的心跳节拍，每天一次会让 webhook 重试、token 清理、deletion sweep 全部退化到日级粒度。**这是当前最严重的生产语义偏差**，必须立刻确认意图并对齐。

3. **`runWorkerTick` 的 `LOCK_TIMEOUT` 无自愈监控**
   `src/services/jobs/runner.ts` 靠 `lockedAt < now() - 5min` 翻 RUNNING 回 PENDING，但若连续两次 tick 都因为 route 超时被 Vercel kill，`lockedAt` 会被新 worker 不断刷新，job 永远卡 RUNNING。建议 `/api/metrics` 暴露一个 `kitora_jobs_running_too_long` gauge，配合 admin/jobs 的 DLQ 看板做告警。

### P1 — 一般

4. **`src/middleware.ts` 与 `src/lib/region.ts` 的"双轨 region 解析"**
   两边逻辑当前一致（已核对），但在 edge runtime 不能 import Node-only 模块的现实下，未来任何一次解析规则变更都依赖人工同步。建议提取一个**零依赖**的 `src/lib/region-parse.ts` 纯函数，两边都 import；`region.ts` 在外面加缓存与 logger，`middleware.ts` 直接调用纯函数。这是 RFC 0005 留下的最容易踩坑的入口。

5. **认证状态字段过多，状态机隐式**
   `User.status` × `User.twoFactorEnabled` × `User.sessionVersion` × `User.deletionScheduledAt` × JWT.`tfa_pending` × `userStatus` × `userRegion`，组合数 32+，分散在 middleware + jwt callback + 各个 server action 里 if/else 推。目前还能 hold，但下一个新增维度（比如"邮箱待验证才许只读"）就会破。建议引入 `xstate` 或一份 `computeAuthGate(user, jwt) → { canAccess, redirectTo, reason }` 单一计算函数。

6. **软删除策略不统一**
   `Membership.deletedAt`（SCIM reactivate 用）与 `User.status='PENDING_DELETION'` + 30 天硬删（GDPR 用）两种，**所有 list 查询都要手动写过滤**。建议在 `src/services/orgs/queries.ts` 里集中一个 `listActiveMembers(orgId)`、`countActiveUsers()`，禁止业务代码直接 `prisma.membership.findMany`。

7. **Region × Provider 的 factory 没有"未实现就硬错"防线**
   EU region 在 `src/lib/region/providers.ts` 多数走 fallback 到 GLOBAL provider，意外在 EU 部署上就会创建 Stripe customer。建议每个未实装组合 `throw new Error('region-not-implemented: EU/billing')`，把"占位"变成"显式不可用"。

8. **三份 `docker-compose*.yml` 维护成本**
   `docker-compose.yml` / `docker-compose.cn.yml` / `docker-compose.eu.yml` 大段重复，靠人工 diff 同步。建议拆 `docker-compose.base.yml` + region overlay，或干脆只保留 GLOBAL 一份 + Aliyun ACK YAML（CN）+ EU 占位文档说明，删掉 EU compose（既然没人用）。

### P2 — 轻微

9. `prisma/schema.prisma` 的 `BackgroundJob` 表已经塞了 priority + queue 字段但 **v1 完全不用**（注释明说），属于过度前瞻；不严重，但提示"先要再说"原则可以再坚定一点。
10. `/api/metrics` 路径只校验 ADMIN bearer 但**未限流**，配合 8 个 `prisma.count/groupBy`，可能成为放大攻击靶。建议套 `apiLimiter` 或 IP 级 token-bucket。
11. `infra/aliyun/main.tf` 与 `docker-compose.cn.yml` 各自维护一套 CN 资源描述，没有 cross-check；首次 ACK 部署若变量漂了不容易在 CI 时抓出。

## 四、文档与代码同步度

### 重大不同步

1. **`vercel.json` cron schedule = `0 0 * * *`，但 `CHANGELOG.md`、`docs/deploy/global.md:83` 写 `* * * * *`**
   见 §三/2。**生产行为偏差**，立即对齐。

2. **`docs/rfcs/0008-background-jobs.md:5` 状态仍是 `Draft`**
   v0.9.0 已完整交付且 §11 实施完成段已回填。改一行即可。同步确认 0001–0007 的状态字段是否也都翻成 Implemented。

3. **`README.md:170-184` 的"项目结构"图严重过时**
   只列了 `src/lib/{auth,stripe,email,db,logger,analytics,rate-limit,request,utils}`，**完全没有** `jobs / orgs / webhooks / sso / webauthn / data-export / storage / region / billing / account / admin` 这 11 个子目录——也就是 RFC 0001/0002/0003/0004/0006/0007/0008 全部隐形。新人按 README 看到的是一个 v0.4 的项目。

4. **`README.md:46-57` 技术栈表只列 Stripe/Resend/Vercel**
   与第 9-27 行的"CN 区工程层完成（Aliyun OSS / DirectMail / Alipay+WeChat / ioredis）"自相矛盾。建议改成 region × provider 矩阵。

### 中等不同步

5. **`CHANGELOG.md:42` 写 admin/jobs/page.tsx "481 行"，实际 483 行**
   Changelog 里抄了具体行数，每改一次就要跟着同步，性价比偏低；建议未来用"~480 行"或干脆不写行数。

6. **`README.md:206-230` 的开发计划只到 v0.7.x（CN region + RFC 0001 + RFC 0002）**
   没有 RFC 0003 webhooks、0004 SSO、0007 WebAuthn、0008 background jobs 的对勾，与实际进度脱节 4 个版本。

7. **`README.md:46-57` 的"部署"层写 Vercel**，但项目同时维护 Dockerfile + ACK + Fly Machines 文档；新人会以为只有 Vercel 一条路。

### 轻微/笔误

8. `README.md:184` 提到 `src/middleware.ts`，实际就在那里，结构正确——但"国际化路由 / Auth.js 中间件"两件事的顺序与代码内调用顺序相反，容易让人误解。
9. `docs/rfcs/README.md`（活索引表）需要核对各 RFC 状态列是否都更新到了 Implemented。
10. `openapi/v1.yaml` 里 `/api/jobs/tick` 是内部端点，**故意不收录**（CHANGELOG 明确说明），这点 OK；但 `pnpm openapi:check` 脚本是否真的覆盖了所有 v1 endpoint 与 yaml 的双向一致，建议 CI 跑一次确认。

## 五、修复优先级（建议两周内闭环）

第一周（保命）：先把 `vercel.json` cron 修对、Stripe webhook 顺序调整、middleware/region 解析提纯函数、RFC 0008 状态翻 Implemented、README 项目结构与技术栈刷新——这五项加起来 1–1.5 个人天，但解决了"线上行为不符预期"和"新人第一印象错位"两个最伤的问题。

第二周（收口）：把 jwt callback 的状态机抽出去、补 billing/sso/webhooks 三块的单测、把 `/api/jobs/tick` 加 Vercel 计划探测、统一软删除查询入口。再做一次"`pnpm openapi:check` + `pnpm egress:check` + `pnpm typecheck` + `pnpm test:unit` + `pnpm test:e2e`" 五件套全绿，作为 v0.9.1 的发布门禁。

之后再开 RFC 0009/0010 不迟。当前最大的杠杆点是**让既有能力对外的"可见面"和对内的"可维护面"都恢复一致**，而不是再加新东西。

---

## 六、亮点（值得继承到下一阶段）

- **RFC 驱动 + CHANGELOG 双轨**的开发节奏在小团队里是最优解，唯一缺的是 RFC 落地后回填状态字段的强制流程。
- **`(email, region)` 复合唯一 + `currentRegion()` 单一入口**的多 region 设计干净利落，比"按用户做 sharding key"这种方案省心一个数量级。
- **secret 加密统一走 HKDF(AUTH_SECRET, salt=row.id) + AES-256-GCM**（2FA / webhook / OIDC client secret），这是教科书级实现，建议在新模块继续沿用。
- **`server-only` 的边界控制**到位，没有看到 secret 泄到 client bundle 的迹象。
- **OpenAPI + Scalar + check-coverage + egress audit + commitlint + husky** 整套外围工程化已经形成，比绝大多数同体量项目领先两个身位。

---

> 一句话总结：**Kitora 在能力广度上已经超过它需要的程度，下一阶段的最大投入应该花在收口、瘦身、状态机化与文档对齐上。**
