## 1. 背景与目标

`/api/v1/me` 和 webhook 投递管线已经支撑得起小型 B2B 客户，但要把单合同价格从 «每月几十美元» 推到 «每年五位数 USD»，**SAML SSO 是绕不过去的硬门槛**。理由很机械：500+ 人公司的 IT 安全策略一律要求新接入的 SaaS 走他们的 IdP（Okta / Azure AD / Google Workspace / OneLogin），不接入就直接被采购流程否决。

OIDC 是 SAML 的现代等价物——同样的「身份联邦 + 集中登出」，但用 JSON 而非 XML，发证流程更轻量。两者并行支持是行业惯例，不是过度设计。

SCIM（System for Cross-domain Identity Management）则是 SSO 的「自动化部分」：当 IT 在 Okta 给一个新员工分配 Kitora 应用时，不需要这个员工先来 Kitora 注册——SCIM 端点会预先创建账号 + 分配角色；离职时反向回收。这是 enterprise 客户的「合规底线」，没有 SCIM 就需要 IT 手动 onboarding/offboarding，违反 SOC 2 §AC-2 的可证明控制要求。

**目标**：

- **per-org IdP 配置** —— OWNER/ADMIN 在 `/settings/organization/sso` 上传 SAML metadata XML，或填 OIDC issuer + clientId/secret。一个 org 最多 1 SAML + 1 OIDC（绝大多数客户只用一种）。
- **SP-initiated 登录** —— 登录页加 「Sign in with SSO」，输入邮箱后通过 emailDomains 匹配到 IdP，跳转完成认证。IdP-initiated 也支持，但仅限 SAML（行业惯例）。
- **JIT provisioning** —— 首次 SSO 登录的用户没有 Kitora 账号时，自动创建 + 自动加入对应 org，role = IdP 配置的 `defaultRole`（默认 MEMBER）。
- **SCIM v2 端点** —— `/scim/v2/Users` + `/scim/v2/Groups`，per-IdP Bearer token 鉴权。Group 映射到 OrgRole（Owners / Admins / Members）。
- **强制 SSO** —— `IdentityProvider.enforceForLogin = true` 时，该 org 的成员只能走 IdP 登录，password / OAuth 一律拒绝（API token 不受影响）。
- **与 RFC 0002 的交互** —— SSO 用户**默认豁免 2FA**（IdP 自身负责强 MFA），除非 org 同时勾选了 require2fa（双保险，少见但合理）。

**非目标**：

- **SSO 协议自实现** —— 用 [`@boxyhq/saml-jackson`](https://github.com/boxyhq/jackson) 处理 SAML XML 与签名验证。自写等同于自找苦吃。
- **多 IdP per-org** —— 同时挂 Okta 和 Azure AD 极少见，留 follow-up RFC。
- **SAML SLO（Single Logout）** —— SP-initiated 登出已经满足 99% 场景；IdP-initiated 登出复杂度高、客户实际不用。
- **SCIM 2.0 完整覆盖** —— v1 只实现 enterprise 客户实际用的端点（Users CRUD、Group membership 同步），跳过 `/Schemas`、`/ResourceTypes` 等元数据查询。
- **WebAuthn / Passkey** —— 是 RFC 0006 的事。
- **个人用户的"Sign in with Google/GitHub"** —— Auth.js 已经支持，本 RFC 不动那条路径。

## 2. 数据模型

```prisma
model IdentityProvider {
  id              String       @id @default(cuid())
  orgId           String
  name            String                                   // "Okta production"，UI 上展示
  protocol        SsoProtocol                              // SAML | OIDC

  // ── SAML ──────────────────────────────────────────
  // saml-jackson 需要的最小集合：上传完整 metadata XML，库内自己解析出 SSO URL + signing cert。
  samlMetadata    String?      @db.Text

  // ── OIDC ──────────────────────────────────────────
  oidcIssuer        String?                                // https://accounts.example.com
  oidcClientId      String?
  // AES-256-GCM, HKDF-derived per row id（同 RFC 0002 / 0003 模式）
  oidcClientSecret  Bytes?

  // ── 通用配置 ─────────────────────────────────────
  // 登录页 emailDomains 匹配：用户输入 jane@acme.com，按 "acme.com" 找到该 IdP。
  // 数组而非单值，因为大公司常有多个 owned domains（acme.com + acme.io + acquisition.com）。
  emailDomains    String[]     @db.Text
  defaultRole     OrgRole      @default(MEMBER)

  // 强制走 SSO：true → 该 org 成员的 password / OAuth 登录路径直接拒绝（API token 仍可用）。
  enforceForLogin Boolean      @default(false)
  enabledAt       DateTime?    // null 即"已配置但未生效"，给 OWNER 一个先 dry-run 的选项

  // ── SCIM ──────────────────────────────────────────
  // sha256(plaintext) — 平文仅在 rotate 时一次性返回。同 ApiToken 模式。
  scimTokenHash   String?      @unique
  scimTokenPrefix String?      // 前 8 字节 base64url，便于 UI 识别
  scimEnabled     Boolean      @default(false)

  createdAt       DateTime     @default(now())
  updatedAt       DateTime     @updatedAt

  organization    Organization @relation(fields: [orgId], references: [id], onDelete: Cascade)

  // 一 org 最多 1 SAML + 1 OIDC。同一 protocol 想换配置就 update 现有行。
  @@unique([orgId, protocol])
  @@index([orgId])
  @@index([scimTokenHash])
}

enum SsoProtocol {
  SAML
  OIDC
}

// Membership 加列 —— 关联 IdP 子标识（NameID 或 OIDC sub），跨登录会话稳定。
// 不放在 User 上是因为同一邮箱可以同时是 Acme 的 SAML 用户（外包）和 PersonalOrg 的 password 用户。
model Membership {
  // ... 现有列
  providerSubject String?  // SAML NameID 或 OIDC sub
  providerId      String?  // 引向 IdentityProvider.id

  @@index([providerId, providerSubject])  // SCIM lookup 走这个
}
```

`Organization` 加反向关系 `identityProviders IdentityProvider[]`，最多 2 行。

## 3. 登录流程

### 3.1 SP-initiated（90% 场景）

```
[user]                    [Kitora]                    [IdP]
  │                          │                          │
  │  GET /login              │                          │
  │ ◄────────────────────── │                          │
  │  Submit "Sign in with    │                          │
  │  SSO" + email            │                          │
  │ ─────────────────────► │                          │
  │                          │ 解析 email domain        │
  │                          │ → IdentityProvider 行    │
  │                          │ → 构造 SAML AuthnRequest │
  │  302 → IdP SSO URL +    │ │ │   RelayState=...
  │ ◄──────────────────────│                          │
  │ ─────────────────────────────────────────────► │
  │                          │                          │ 用户在 IdP 上
  │                          │                          │ 完成认证
  │ ◄─────────────────────────────────────────────── │
  │  POST /api/auth/sso/saml/[providerId]/callback     │
  │ ─────────────────────► │                          │
  │                          │ saml-jackson 验签        │
  │                          │ → 找/建 user             │
  │                          │ → 找/建 membership       │
  │                          │ → 写 DeviceSession       │
  │                          │ → 设 session cookie      │
  │  302 → /dashboard       │                          │
  │ ◄──────────────────────│                          │
```

OIDC 流程同构，只是 AuthnRequest 换成 OIDC `authorize` redirect，回调换成 `code → token` exchange。

### 3.2 IdP-initiated（仅 SAML）

用户从 IdP 应用市场点 Kitora tile：IdP 直接 POST AuthnResponse 到 `/api/auth/sso/saml/[providerId]/callback`，没有 RelayState。我们落到默认 `/dashboard`。

OIDC 不支持 IdP-initiated（协议本身没这玩意），所以这部分逻辑只在 SAML 路径。

### 3.3 JIT 用户与 Membership

回调拿到 NameID / sub 后：

1. `Membership.findFirst({ providerId, providerSubject })`：找到就直接登录。
2. 没找到时按 NameIDFormat（通常是 emailAddress）反查 `User.email`。找到 → 创建 Membership（role = `defaultRole`）+ 写 providerId/providerSubject。
3. 还是没找到 → 创建新 User + Membership（自动 emailVerified=now，因为 IdP 已验证）。

`providerSubject` 的稳定性是关键——IT 把员工 email 从 jane@acme.com 改成 j.smith@acme.com 时，IdP 的 NameID 不变，我们就能识别出"还是同一人"，避免重复账号。

### 3.4 与 password / 2FA / OAuth 的互斥

```ts
// 简化伪代码
async function authorizeCredentials({ email, password }) {
  const user = await prisma.user.findUnique({ where: { email } });
  if (!user) return null;

  // RFC 0004：检查这个 user 所在的任何 org 是否启用了 enforceForLogin
  const enforcing = await prisma.identityProvider.findFirst({
    where: {
      enforceForLogin: true,
      enabledAt: { not: null },
      organization: { memberships: { some: { userId: user.id } } },
    },
    select: { id: true, organization: { select: { slug: true } } },
  });
  if (enforcing) {
    throw new SsoRequiredError(enforcing.organization.slug);
  }

  // 现有 bcrypt 比对...
}
```

UI 展示：当用户输入的 email 命中某个 enforceForLogin = true 的 org，登录卡片自动隐藏密码框，只显示 「Continue with SSO」按钮。这避免了"输错密码 8 次后才知道这家公司必须走 SSO"的体验。

2FA 互斥：SSO 路径的回调里直接设 `tfa: 'verified'`，跳过 RFC 0002 的 tfa_pending 状态机——IdP 的 MFA 已经把好门了。除非 org 同时开了 `require2fa`（极少见），那这个 user 即便走完 SSO 还得在 Kitora 这边再过一次 TOTP。

## 4. SCIM v2 端点

`/scim/v2/*` 路由，挂在 `/api/scim/v2/*`（避免和 i18n routing 冲突）。Bearer 鉴权用 `IdentityProvider.scimTokenHash`：

```bash
curl -H "Authorization: Bearer scim_..." \
     https://app.kitora.com/api/scim/v2/Users
```

### 4.1 v1 必须实现的端点

```
GET    /api/scim/v2/Users                    # 列表 + filter（仅支持 userName eq + externalId eq）
POST   /api/scim/v2/Users                    # 创建（IT 给员工分配应用）
GET    /api/scim/v2/Users/{id}
PATCH  /api/scim/v2/Users/{id}               # 改 active=false → 禁用 membership（不删 user，方便恢复）
DELETE /api/scim/v2/Users/{id}               # 真·移除 membership
GET    /api/scim/v2/Groups                   # 三个固定组：Owners / Admins / Members
GET    /api/scim/v2/Groups/{id}
PATCH  /api/scim/v2/Groups/{id}              # 改组成员 = 改 role
GET    /api/scim/v2/ServiceProviderConfig    # IdP discovery 必备
GET    /api/scim/v2/Schemas                  # 同上，返回固定 schema
```

### 4.2 v1 不实现

- `/Users/{id}/Schemas/*`、`/ResourceTypes` —— 仅极少数 IdP 会查
- `eq` 之外的过滤算子 —— Okta / Azure 实测都只用 eq

### 4.3 disable vs delete

SCIM PATCH `active: false` 在 IdP 那边表示"先停用，可能还会激活"——我们对应到 `Membership.deletedAt` 软删（新增列）。SCIM DELETE 对应硬删 Membership 行。User 行不动——同一人在另一个 org 还可能有 Membership。

### 4.4 Group 映射

固定三个 group，与 OrgRole 一一映射：

| SCIM Group displayName | Kitora OrgRole |
| ---------------------- | -------------- |
| Owners                 | OWNER          |
| Admins                 | ADMIN          |
| Members                | MEMBER         |

OWNER 不能通过 SCIM 添加（避免 IT 误操作把全公司变成 OWNER），需要在 Kitora 内手动 promote。SCIM PATCH 试图加 OWNER 时返回 400 + `{ "scimType": "noTarget" }`。

## 5. UI

### 5.1 OWNER/ADMIN 配置页 `/settings/organization/sso`

- IdentityProvider 列表（最多 2 行）
- 每行展开后能看到 protocol / status / emailDomains / SCIM token prefix
- "Add SAML provider" / "Add OIDC provider" 按钮
- SAML 配置 form：粘贴 metadata XML（必填）+ emailDomains（多选）+ defaultRole + enforceForLogin checkbox
- OIDC 配置 form：issuer + clientId + clientSecret + emailDomains + ...
- "Generate SCIM token" 按钮：创建并一次性返回 `scim_<base64url(32)>`，写 sha256 + prefix
- "Test SSO" 按钮：触发一次 dry-run 登录，断言 metadata 解析成功 + 用户被路由到 IdP，但回调直接返回 200 而不真的设 session（仅做 e2e 用）

### 5.2 登录页

```
┌─────────────────────────────────────┐
│  Sign in to Kitora                  │
│                                     │
│  Email   [_______________________]  │
│  Password [_______________________] │
│                                     │
│  [ Sign in ]                        │
│                                     │
│  ─── or ───                         │
│                                     │
│  [ Continue with SSO ]              │
└─────────────────────────────────────┘
```

「Continue with SSO」点击后弹一个 inline form，只输入 email，提交后按 emailDomains 路由。如果 user 输入的 email 命中 enforceForLogin = true 的 org，密码栏自动 disable + 提示「Your organization requires SSO」。

## 7. 迁移计划（拆 4 个 PR）

### PR-1 Schema + IdP CRUD UI（无登录流）

- 加表 + 枚举 + audit codes + Membership 列。
- Server actions：create / update / delete IdP，rotateScimToken。
- UI：`/settings/organization/sso` 页 + 表单（OWNER/ADMIN）。
- 公开 API：`/api/v1/orgs/{slug}/identity-providers` CRUD（带 OpenAPI 同步）。
- 不接登录流——这个 PR 落地时 SSO 入口在 UI 上还看不到。
- e2e：建 SAML provider → list → rotate SCIM token（明文一次性可见）→ delete。

### PR-2 SAML 接入 + 登录流

- 新增依赖 `@boxyhq/saml-jackson`（hosted in-process，不部署独立服务）。
- 路由：
  - `POST /api/auth/sso/saml/start` —— 接收 email，构造 AuthnRequest 跳 IdP
  - `POST /api/auth/sso/saml/[providerId]/callback` —— 验签 + JIT + 设 session
- enforceForLogin 检查接入 `authorize()` 凭据回调。
- Login 页加 「Continue with SSO」入口 + 域名嗅探。
- e2e：用 saml-jackson 提供的"mock IdP"（库自带）跑全链路 → 断言 session 设上。

### PR-3 OIDC 接入

- 路由结构与 SAML 镜像：`/api/auth/sso/oidc/start` + `/api/auth/sso/oidc/[providerId]/callback`。
- OIDC client secret 用 RFC 0003 同款 HKDF + AES-256-GCM 落库。
- Auth.js v5 的 OIDC provider 只能静态注册——动态 per-org provider 走自写小客户端（`openid-client` lib）。
- e2e：用 `oidc-provider` 起本地 IdP，与 SAML 测试同形态。

### PR-4 SCIM v2 端点 + 自动 provisioning

- 路由：`/api/scim/v2/Users{,/[id]}`、`/api/scim/v2/Groups{,/[id]}`、`/api/scim/v2/ServiceProviderConfig`、`/api/scim/v2/Schemas`。
- Bearer token 鉴权（lookup `scimTokenHash`）。
- PATCH/DELETE 写 audit row。
- IdP 厂商兼容性测试矩阵：Okta、Azure AD、Google Workspace SCIM 各一份 fixture（本地 mock 即可）。
- 文档站补充：SAML metadata 上传步骤 / SCIM 端点完整 spec / 主流 IdP 配置截图。

回滚：每个 PR 都是加法。PR-2 引入 saml-jackson 依赖——回滚需 revert 加包 + 数据迁移（IdentityProvider 行可保留）。

## 9. 风险与对策

| 风险                                                  | 对策                                                                                                                     |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 上传错的 SAML metadata 把整个 org 锁外                | "Test SSO" dry-run 按钮，启用前必须跑通；enabledAt = null 时配置可见但登录入口不显示                                     |
| OIDC client secret 泄露                               | 仅存 AES-GCM 密文（RFC 0003 同模式）；UI 上 secret 明文仅在 rotate 时一次性返回                                          |
| SCIM token 泄露 → 攻击者批量创建 OWNER                | OWNER 不能通过 SCIM 创建（§4.4 约束）；token rotate 路径一键失效                                                         |
| JIT provisioning 误开 → 任何 IdP 用户进任何 org       | JIT 仅在该 IdP 配置的 emailDomains 命中时触发，且 Org 只有 1 个 IdP（unique 约束），不存在跨 org 误判                    |
| IdP 跑路 / 续费失败 → enforceForLogin 把 OWNER 也锁外 | OWNER 一律保留 password 后门（per-membership flag `bypassSso`）：`Membership.role = OWNER` 时 enforce 不生效             |
| SAML XML 解析依赖 saml-jackson 漏洞                   | 锁版本 + Renovate 周更；saml-jackson 是 BoxyHQ 主推产品，活跃维护，Stripe / Vercel / Linear 都在用                       |
| 加密 secret 跨 AUTH_SECRET 轮换失效                   | 同 RFC 0003：rotate 路径要求重新填一次 OIDC client secret；提供 `bin/rotate-sso-keys.ts` 在 AUTH_SECRET 切换时批量重加密 |
| SCIM 操作 race（并发添加同 user 到多 group）          | Group PATCH 走 `updateMany` + version 字段（OCC），冲突返回 409 让 IdP 重试                                              |
| 中国区数据驻留                                        | `IdentityProvider.region` 占位列（RFC 0005 接入），v1 仅 global                                                          |
| 与 RFC 0002 require2fa 双重启用导致用户体验卡顿       | 文档明示双开是可选；登录回调里检测到 require2fa 时仍走完整 tfa_pending 流程，不静默吞                                    |

## 11. 评审决策（待定稿，2026-04-26 起草）

- [ ] **SAML 库选型** —— 推荐 `@boxyhq/saml-jackson`（Stripe 等大厂在用，BoxyHQ 主推，hosted in-process）。备选 `samlify`（更轻量，但 JIT 接入要自写更多 glue）。
- [ ] **OIDC 客户端** —— 推荐 `openid-client`（IETF 维护，Auth.js 内部就在用）。
- [ ] **JIT 默认行为** —— 推荐"启用，role = MEMBER"。enterprise 客户最常见预期；不喜欢的 IT 可在 IdP 配置里关掉应用分配开关，等价于关 JIT。
- [ ] **enforceForLogin 兜底** —— 推荐"OWNER 不受 enforce 限制"，避免 IdP 故障锁外。
- [ ] **SCIM 端点鉴权** —— 推荐 per-IdP token（不复用 ApiToken）。SCIM 操作和普通 API 调用的访问模式 / 限流策略不同（SCIM 可能是夜间批量），单独 token 隔离更好。
- [ ] **SP-initiated email lookup** —— 推荐"输入 email → 按 emailDomains 解析到 IdP"。备选"输入 org slug"，但用户体验差（要记 slug 而不是 email）。
- [ ] **SLO** —— v1 不做。占位 follow-up RFC。

待用户确认后正式落地为「✓ 已定稿」。
